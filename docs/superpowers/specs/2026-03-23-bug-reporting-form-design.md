# 缺陷录入表单优化设计

## 背景
当前缺陷录入流程已经支持基础字段、首次证据上传和后续补充证据，但实际使用中存在两个明显问题：

- `复现步骤` 输入区过小，不适合填写较长的排查说明与复现路径。
- 缺陷信息被拆分为 `复现步骤`、`环境信息`、`期望结果`、`实际结果` 等多个小输入框，填写时上下切换频繁，不符合真实提单习惯。

用户希望把缺陷编辑体验调整为更接近“长文档式录入”：

- 使用一个默认更大的主文本区。
- 通过内置模板一次性填写环境、步骤、期望结果和补充说明。
- 创建缺陷时可直接上传图片证据。
- 编辑缺陷时也能直接补充图片证据，而不是依赖单独弹窗。

## 目标
- 提升长文本缺陷录入效率，减少表单字段切换。
- 让缺陷创建和缺陷编辑都具备更强的证据补录能力。
- 尽量复用现有后端接口，避免扩大数据库和 API 变更范围。

## 非目标
- 不重做缺陷详情页的信息展示结构。
- 不新增图片预览、拖拽上传、排序等增强能力。
- 不调整后端 `Bug` 模型字段结构。
- 不同步改造“创建工作项”弹窗里的缺陷分支；本次仅覆盖独立的 `新建缺陷` 与 `编辑缺陷` 弹窗，避免范围继续扩大。

## 设计概览

### 方案结论
采用“单主文本区 + 证据区内嵌”的方案：

- `新建缺陷` 弹窗中，将 `复现步骤` 升级为大文本区，提供默认模板。
- `编辑缺陷` 弹窗整体加宽加高，并将 `复现步骤` 同样升级为大文本区。
- 原有 `环境信息`、`期望结果`、`实际结果` 输入框从创建/编辑表单中移除，提示用户统一写入模板化大文本区。
- `新建缺陷` 保留现有首次证据区。
- `编辑缺陷` 在详情编辑表单中新增内嵌证据补充区，复用现有证据上传接口。

### 模板内容
主文本区默认模板如下：

```text
【复现环境】
浏览器/系统：
账号：
版本：

【复现步骤】
1.
2.
3.

【实际结果】


【期望结果】


【补充说明】

```

模板目标：

- 降低提单门槛，避免用户思考“该填在哪个字段”。
- 保证缺陷信息结构一致，便于后续阅读。
- 兼容已有 `steps_to_reproduce` 字段，不引入新的后端字段。

## 界面设计

### 新建缺陷
- 弹窗主体维持两段式结构：`基础信息` 与 `首次证据`。
- `复现步骤` 改为主输入区：
  - 默认高度明显增加。
  - 允许用户继续拖拽拉伸。
  - 初始值直接带出模板，允许用户按模板覆盖填写。
- 移除独立的 `环境信息`、`期望结果`、`实际结果` 字段。
- 在主输入区下增加一段轻提示，说明“环境、步骤、期望结果等请统一填写在该模板中”。
- `首次证据` 区保持在当前弹窗中，继续支持说明、堆栈、截图上传。

### 编辑缺陷
- 弹窗整体最大宽度和最大高度上调，优先适配长文本编辑。
- 详情页签中的 `复现步骤` 升级为与新建页一致的大文本区。
- 移除独立的 `环境信息`、`期望结果`、`实际结果` 编辑框。
- 在表单底部新增 `补充证据` 区：
  - 补充说明
  - 异常堆栈
  - 图片上传
- 保存编辑时分两段提交：
  - 先提交缺陷基础信息。
  - 若证据区有输入，再调用现有证据接口提交补充证据。
- 若证据上传失败：
  - 编辑弹窗保持打开。
  - 已保存的主体内容不回滚。
  - 按钮恢复可点击，便于用户修正后立即重试。

### 现有双入口处理策略
当前仓库中还有“创建工作项”弹窗的缺陷创建分支，它与独立的 `新建缺陷` 不是同一套实现。本次设计明确只优化独立的 `新建缺陷` / `编辑缺陷` 流程，不处理该入口。

原因：

- 用户当前明确指定了独立创建与编辑体验。
- 第二入口尚未支持首次证据两段式提交，纳入本次会扩大改动范围。
- 后续如要统一体验，可再单独做一次缺陷创建入口收敛。

## 数据与兼容策略

### 字段使用
- `title`：继续作为缺陷标题。
- `description`：继续作为缺陷简要描述。
- `steps_to_reproduce`：承载模板化的完整长文本内容。
- `environment` / `expected_result` / `actual_result`：
  - 创建/编辑表单不再暴露。
  - 历史数据保留，不做迁移。
  - 详情展示可继续兼容旧数据。

### 模板注入规则
- `新建缺陷`：初始值直接使用模板。
- `编辑缺陷`：
  - 若 `steps_to_reproduce` 已有内容，则原样展示，不强行覆盖。
  - 若为空，则填入模板，帮助用户开始编辑。
- `environment` / `expected_result` / `actual_result` 的历史值不主动清空；由于表单不再提交这些键，后端继续保留旧数据。

### 历史三字段的编辑态策略
当历史缺陷存在 `environment`、`expected_result`、`actual_result`，但 `steps_to_reproduce` 为空时：

- 编辑弹窗中的主文本区仍使用默认模板。
- 在主文本区上方增加一段只读提示，提醒该缺陷存在历史结构化信息，用户可参考详情页内容后手动整理进主文本区。
- 本次不做自动拼接或自动迁移，避免未经用户确认就改写原始内容。

### 兼容历史数据
- 旧缺陷仍可继续展示已有 `环境信息`、`期望结果`、`实际结果`。
- 新缺陷将主要依赖模板化长文本，不要求后端拆分这些字段。
- 这样可以避免数据库迁移和额外解析逻辑。

## 提交流程

### 新建缺陷
1. 用户填写标题、简要描述、模板化长文本。
2. 用户可选择填写首次证据说明、堆栈、截图。
3. 前端先调用 `POST /api/projects/<project_id>/bugs` 创建缺陷。
4. 若存在首次证据，再调用 `POST /api/bugs/<bug_id>/evidences` 上传证据。
5. 若缺陷创建成功但首次证据上传失败：
   - 提示“缺陷已创建，但首次证据保存失败，请稍后补充”。
   - 关闭当前新建弹窗并进入缺陷列表，避免重复创建主单。
   - 用户后续通过缺陷详情页或编辑页继续补充证据。

### 编辑缺陷
1. 用户修改标题、简要描述、模板化长文本。
2. 用户可在同一界面追加证据说明、堆栈、截图。
3. 前端先调用 `PUT /api/bugs/<bug_id>` 保存缺陷主体信息。
4. 若补充证据区存在有效输入（说明非空、堆栈非空或至少一张图片），再调用 `POST /api/bugs/<bug_id>/evidences` 提交证据。

### 编辑页实现约束
由于缺陷主体更新接口使用 JSON，而证据上传接口使用 `multipart/form-data`，编辑页不能继续沿用“整表单一次提交”的处理方式。

实现约束如下：

- 同一个保存动作内部，前端需显式拆分为两段请求。
- 缺陷主体字段组装为普通 JSON，调用 `PUT /api/bugs/<bug_id>`。
- 证据字段单独组装为 `FormData`，调用 `POST /api/bugs/<bug_id>/evidences`。
- 若证据区为空，则仅发送主体更新请求。
- 发送 `PUT /api/bugs/<bug_id>` 时，请求体必须显式省略 `environment`、`expected_result`、`actual_result` 三个键，禁止传空串或 `null`，避免覆盖历史数据。

## 错误处理
- 若缺陷主体保存失败：
  - 保持弹窗打开。
  - 恢复保存按钮状态。
  - 提示保存失败原因。
- 若新建缺陷主体保存成功，但首次证据上传失败：
  - 提示“缺陷已创建，但首次证据保存失败，请稍后补充”。
  - 不回滚已创建的缺陷主体。
  - 关闭新建弹窗并进入缺陷列表，避免用户误以为需要重新建单。
- 若缺陷主体保存成功，但证据上传失败：
  - 提示“缺陷已保存，但证据提交失败，请稍后补充”。
  - 避免误导用户认为全部失败。
- 若证据区为空：
  - 不触发证据接口调用。
- 若证据区有输入但不满足后端规则：
  - 保持弹窗打开。
  - 展示后端返回的校验错误。
  - 规则沿用现有接口限制：最多 5 张、单张最大 5MB、仅支持图片。

## 测试策略

### 自动化测试
- 新增或更新前端脚本测试，至少覆盖：
  - 新建缺陷弹窗包含模板化大文本区。
  - 编辑缺陷弹窗包含更大的主文本区与内嵌证据区。
  - 现有缺陷详情弹窗的转义逻辑不被破坏。
- 若现有脚本测试难以覆盖文件上传流程，则以关键结构测试 + 手工回归验证组合完成本次验收。

### 手动验证
- 新建缺陷时模板是否默认出现。
- 新建缺陷时是否能成功上传首次图片证据。
- 编辑缺陷时是否能直接追加证据，无需跳转单独弹窗。
- 编辑旧缺陷时，已有 `steps_to_reproduce` 是否被保留。
- 编辑旧缺陷时，历史 `环境信息/期望结果/实际结果` 是否仍能在详情页查看。
- 编辑旧缺陷时，如果仅存在历史 `环境信息/期望结果/实际结果` 而 `steps_to_reproduce` 为空，编辑页是否出现兼容提示且保存后不误清空历史字段。

## 风险与权衡
- 将多个字段合并到单一长文本区后，结构化检索能力会变弱，但换来更高的录入效率和更符合实际使用习惯的体验。
- 由于列表搜索当前仅匹配 `title` 与 `description`，写入 `steps_to_reproduce` 的模板内容默认不会被搜索命中，这是本次接受的范围内权衡。
- 详情页会进入“新旧数据混合展示”的过渡阶段，但该策略能避免数据库迁移和后端兼容成本。
- 编辑页内嵌证据提交流程会增加前端处理分支，但后端接口保持不变，整体风险可控。
