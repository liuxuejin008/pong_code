# 完整功能集成测试设计

## 背景
当前仓库已经具备一套可运行的 Flask + 原生 JavaScript 单页应用，业务覆盖认证、组织、团队、项目、迭代、看板、需求、任务和缺陷等模块，但自动化测试仍以局部功能为主，尚未形成一套覆盖核心业务闭环的完整功能测试体系。

现阶段测试文件主要集中在最近新增的缺陷证据与表单逻辑上，缺少以下能力：

- 从真实用户视角验证各业务模块是否能串联跑通。
- 覆盖多人协作链路，尤其是“组织创建者创建业务数据，其他成员按组织名称加入后可见组织下项目及下游数据”的场景。
- 形成可重复执行的本地浏览器集成回归测试。

用户这次的目标不是补单元测试，而是针对当前项目做一轮完整的功能测试设计，并以浏览器集成测试为主验证主流程和常见异常分支。

## 目标
- 为当前项目定义一套完整的集成功能测试范围与验收口径。
- 先整理完整测试用例，再落地核心业务链路的浏览器集成测试。
- 覆盖单人创建流程与多人协作加入流程。
- 优先保证主流程和常见异常分支可验证、可复跑。

## 非目标
- 不在本轮扩写单元测试。
- 不做性能压测、跨浏览器兼容性测试和像素级 UI 检查。
- 不追求一次性把全部细枝末节场景都自动化。
- 不在本轮实现新的业务功能或改造业务流程。

## 设计概览

### 方案结论
采用“完整测试用例先行 + Playwright 本地浏览器集成测试执行”的方案：

- 先输出覆盖全部核心模块的完整测试用例，作为本轮测试基线。
- 再选择高价值链路编写浏览器集成测试脚本，验证真实页面、真实接口和真实数据联动。
- 对自动化投入回报低但仍重要的场景，保留在手工测试清单中。

### 方案对比

#### 方案一：纯手工功能测试
优点：

- 启动快。
- 短期内容易覆盖较多页面。

缺点：

- 难以复跑。
- 后续回归成本高。
- 很难稳定覆盖多人协作与跨模块联动。

#### 方案二：完整测试用例 + 核心集成自动化
优点：

- 既能保证覆盖完整性，也能沉淀可复跑回归脚本。
- 能优先解决高风险链路的验证问题。
- 适合当前仓库已有少量测试、但尚无完整端到端体系的阶段。

缺点：

- 前期需要先梳理测试矩阵。
- 需要控制自动化范围，避免一次做得过重。

#### 方案三：直接全量自动化
优点：

- 长期价值最高。

缺点：

- 前期投入过大。
- 容易被环境、数据隔离、选择器稳定性等问题拖慢。
- 不适合当前先建立测试基线的目标。

### 方案推荐
推荐采用方案二，即“完整测试用例 + 核心集成自动化”。

原因：

- 与当前需求最匹配：先整理完整测试用例，再用浏览器实际跑一遍关键功能。
- 能兼顾测试范围完整性和执行落地性。
- 可以把近期高风险区域（如缺陷证据上传）与多人协作链路纳入优先回归范围。

## 测试范围与覆盖原则

### 范围定义
本轮测试定义为：针对当前本地项目执行一轮完整的集成功能测试，覆盖所有核心业务模块的主流程，以及每个模块最常见、最容易出现回归问题的异常分支。

### 覆盖原则
- 以浏览器集成测试为主，从用户视角验证前后端联动。
- 每个模块至少覆盖 1 条主流程。
- 每个模块补充 1 到 3 条高频异常分支。
- 优先验证跨模块串联关系，而不是只验证孤立页面。
- 保留现有局部测试，但不把它们视作本轮完整功能测试的主要交付。

### 模块范围
本轮纳入以下模块：

1. 认证
2. 组织
3. 团队
4. 项目
5. 迭代
6. 看板
7. 需求
8. 任务 Issue
9. 缺陷 Bug

### 不重点覆盖的内容
- 性能与压力测试。
- 跨浏览器兼容性测试。
- 像素级视觉回归。
- 纯文案逐字校验。
- 单元测试扩写。

## 测试分层与执行方式

### 第一层：完整测试用例层
这一层负责定义“应该测什么”，覆盖：

- 主流程用例。
- 常见异常分支。
- 基础校验与空态。
- 关键跳转和数据关联。

这层追求的是覆盖完整性，不要求所有场景自动化。

### 第二层：浏览器集成执行层
这一层负责验证“真实用户操作是否跑通”，重点验证：

- 页面是否正常加载。
- 表单提交是否成功联通后端。
- 创建、编辑、列表、详情是否一致。
- 模块串联是否成立。
- 常见异常提示与页面行为是否合理。

建议使用 Playwright 在本地环境执行。

### 自动化优先原则
优先自动化以下场景：

- 注册、登录、退出。
- 创建组织、团队、项目。
- 创建迭代、需求、任务、缺陷。
- 看板状态流转。
- 缺陷证据上传。
- 组织成员加入后的数据可见性验证。

### 先保留为手工的场景
- 低频管理类操作。
- 短期内选择器不稳定、自动化收益较低的页面细节。
- 某些细粒度文案与轻提示验证。

## 核心业务链路

### 1. 首次使用链路
用户注册 -> 登录 -> 创建组织 -> 可选创建团队 -> 创建项目 -> 进入项目上下文。

验证目标：

- 空用户能否顺利进入可协作状态。
- 基础入口与导航是否正常。

### 2. 组织加入与协作可见性链路
用户 A 注册并登录 -> 创建组织 -> 创建项目与部分业务数据 -> 用户 B 通过“输入组织名称并加入”的现有流程加入组织 -> 用户 B 登录进入组织 -> 验证是否能看到组织下的项目、需求、任务、缺陷、迭代等数据。

这是本轮高优先级必测链路。

验证目标：

- 当前实际的组织加入流程是否真实打通。按现有后端实现，加入组织使用 `POST /api/organizations/join`，由已登录用户提交组织名称完成自助加入，而不是邀请链接或管理员定向拉人。
- 成员加入后组织上下文是否正确生效。
- 项目可见性与下游业务数据可见性是否一致。
- 不存在“组织可见但项目或下游数据不可见”的断层。

建议补充的异常分支：

- 未加入组织的用户不能看到对应数据。
- 重复加入、组织名称为空、组织不存在等异常应有明确反馈。
- 加入成功后刷新页面仍应看到正确数据。
- 成员若权限不足，应只具备合理的查看/协作能力，不应拥有管理员级操作。

### 3. 迭代规划链路
进入项目 -> 创建迭代 -> 查看迭代列表 -> 校验状态与展示。

### 4. 需求管理链路
进入需求页 -> 创建需求 -> 列表中可见 -> 搜索需求 -> 更新需求状态。

### 5. 任务协作链路
创建任务 Issue -> 指定负责人与迭代 -> 进入看板 -> 移动任务状态 -> 返回列表确认结果。

### 6. 缺陷处理链路
创建缺陷 -> 编辑缺陷 -> 补充证据/截图 -> 查看证据时间线 -> 记录工时。

## 测试数据策略与前置条件

### 测试账号设计
建议至少准备以下账号角色：

1. 用户 A：组织拥有者/创建者。
2. 用户 B：普通组织成员。
3. 用户 C：未加入组织的对照用户（可选）。

### 基础测试数据
每次执行测试，建议至少生成一套完整数据：

- 1 个组织
- 1 个团队
- 1 个项目
- 1 个迭代
- 1 到 2 条需求
- 1 到 2 条任务
- 1 条缺陷
- 1 份缺陷证据附件

### 数据命名策略
所有测试数据带唯一标识，建议包含时间戳或随机后缀，例如：

- `测试组织-时间戳`
- `集成测试项目-时间戳`
- `需求A-时间戳`
- `缺陷A-时间戳`

### 前置条件
在正式执行浏览器集成测试前，需要确认：

1. 本地服务可正常启动，首页可访问。
2. 数据库可用于重复创建测试数据，最好使用专用测试库或可清理的本地库。
3. 上传目录可写，缺陷证据上传功能可用。
4. 测试脚本可以独立创建账号与数据，尽量减少人工预置。
5. 当前产品的组织加入方式已确认，并能在测试中稳定触发。

### 数据隔离原则
- 每次执行尽量创建独立数据，不依赖历史脏数据。
- 失败场景不能污染后续主流程。
- 多用户场景要明确谁创建、谁加入、谁验证。
- 若本地已有大量旧数据，执行前应清理或切换专用测试库。

## 权限矩阵
当前测试设计需要与现有后端权限模型对齐。根据当前接口实现：

- 创建组织时，创建者会自动成为该组织 `admin`。
- 普通加入组织的用户角色为 `member`。
- 项目访问以“是否为组织成员”为准。
- 创建项目和创建团队要求组织 `owner` 或 `admin`，普通 `member` 不允许。
- 加入团队要求用户先是组织成员。

建议把以下权限差异直接纳入测试用例与自动化断言：

| 角色 | 查看组织详情/成员 | 查看项目/需求/任务/缺陷/迭代 | 创建项目 | 创建团队 | 加入团队 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 组织 owner | 允许 | 允许 | 允许 | 允许 | 允许 |
| 组织 admin | 允许 | 允许 | 允许 | 允许 | 允许 |
| 组织 member | 允许 | 允许 | 拒绝 | 拒绝 | 允许 |
| 非组织成员 | 拒绝 | 拒绝 | 拒绝 | 拒绝 | 拒绝 |

权限断言建议至少覆盖：

1. 用户 B 作为新加入的 `member`，能看到组织下项目及其下游数据。
2. 用户 B 作为 `member`，不能创建项目。
3. 用户 B 未加入组织前，不能访问组织详情、项目详情和团队详情。
4. 用户 B 未加入组织前，不能加入该组织下团队；加入组织后可以加入团队。

## 详细测试矩阵与验收标准

### 认证模块
必测点：

- 注册成功。
- 注册必填校验。
- 登录成功。
- 错误账号或密码登录失败。
- 未登录访问受保护资源被拦截。
- 退出登录后登录态失效。

验收标准：

- 成功场景返回正确页面或数据。
- 失败场景有明确反馈。
- 未登录时无法看到业务数据。

### 组织与成员模块
必测点：

- 创建组织成功。
- 组织列表可见。
- 用户 B 成功加入组织。
- 用户 B 加入后能看到组织下项目。
- 用户 B 加入后能看到项目下需求、任务、缺陷、迭代。
- 未加入组织的用户不能看到对应数据。

验收标准：

- 成员加入后可见性立即生效或刷新后生效。
- 不存在组织可见但项目或下游数据不可见的断层。
- 非成员无法通过正常路径访问组织资源。
- 新加入的普通成员能够查看数据，但创建项目类管理操作会被拒绝。

### 团队模块
必测点：

- 创建团队。
- 加入团队或离开团队。
- 团队成员列表展示。
- 组织与团队上下文切换正常。

验收标准：

- 成员变化后展示同步更新。
- 空态正常。
- 重复加入或非法加入有合理反馈。
- 非组织成员加入团队时应被拒绝；加入组织后再次加入团队应成功。

### 项目模块
必测点：

- 在组织下创建项目。
- 项目列表展示。
- 进入项目上下文。
- 新成员加入组织后能看到该项目。

验收标准：

- 项目创建后列表可见。
- 核心字段展示正确。
- 组织成员可见性与项目可见性一致。
- 普通组织成员不可创建项目，页面或接口需明确反馈拒绝结果。

### 迭代模块
必测点：

- 创建迭代。
- 查看迭代列表。
- 校验状态展示。
- 空态展示。
- 项目成员进入后可见已有迭代。

验收标准：

- 迭代创建后可在列表查到。
- 状态展示与后端一致。
- 成员加入后可见性与项目保持一致。

### 需求模块
必测点：

- 创建需求。
- 查看需求列表。
- 搜索需求。
- 更新需求状态。
- 新成员加入后可见已有需求。

验收标准：

- 搜索结果准确。
- 状态变更后列表同步更新。
- 项目成员可见所属项目需求。

### 任务 Issue 模块
必测点：

- 创建任务。
- 编辑任务。
- 分配负责人。
- 关联迭代。
- 状态变化。
- 新成员加入后可见已有任务。

验收标准：

- 编辑后列表与详情一致。
- 关联字段正确回显。
- 成员可见性符合组织/项目权限预期。

### 看板模块
必测点：

- 进入看板页。
- 任务在 `todo / doing / done` 之间流转。
- 刷新后结果仍正确。
- 空看板展示正常。

验收标准：

- 流转结果真实落库。
- 刷新后仍在正确列。
- 新成员加入后可看到相同看板数据。

### 缺陷模块
必测点：

- 创建缺陷。
- 编辑缺陷。
- 查看缺陷详情。
- 添加证据/截图。
- 上传非法文件失败。
- 记录工时。
- 新成员加入后可见已有缺陷。

验收标准：

- 缺陷主体保存正确。
- 证据上传成功后可在详情中看到。
- 非法文件上传不会产生脏数据。
- 工时、证据和详情展示彼此不串字段。

## 执行顺序
建议按以下顺序推进：

1. 整理完整测试用例。
2. 明确自动化优先级。
3. 准备本地测试环境。
4. 编写 Playwright 集成测试脚本。
5. 执行测试并记录结果。
6. 输出测试结论与问题清单。

### 自动化优先链路
建议优先落地以下链路：

1. 注册/登录/创建组织与项目。
2. 组织成员按“输入组织名称加入”流程加入后，验证成员可见项目与下游数据。
3. 创建迭代。
4. 创建需求并搜索。
5. 创建任务并在看板流转。
6. 创建缺陷并补充证据。

## 自动化执行契约
为便于后续直接拆解为 Playwright 脚本，本轮建议固定以下执行契约：

### 运行入口
- 默认基址使用 `http://localhost:5001`，与当前 `app.py` 的本地启动端口保持一致。
- 启动命令默认使用 `python app.py`。
- 若测试时切换端口，需通过统一配置覆盖，不在脚本中硬编码多个地址。

### 环境变量
执行前应显式设置或确认以下变量：

- `DATABASE_URL`：建议指向可重复清理的测试库，不直接复用长期开发库。
- `SECRET_KEY`：测试环境可使用固定测试值。
- `BUG_EVIDENCE_UPLOAD_DIR`：建议指向独立测试目录，便于清理上传文件。

### 多用户会话策略
- 用户 A 与用户 B 建议使用不同浏览器上下文隔离会话。
- 若脚本复杂度受限，也可采用“显式退出 -> 切换账号重新登录”的方式，但优先推荐双上下文。
- 多用户相关断言应明确哪个上下文创建数据、哪个上下文验证可见性与权限。

### 可观测断言约束
后续用例和脚本应尽量使用可量化信号，而不是主观描述。建议优先断言：

- HTTP 状态码或接口成功/失败标识。
- 列表中是否出现指定标题或唯一标识数据。
- 刷新页面后数据是否仍存在。
- 错误提示是否包含约定关键字，如“未找到该组织”“您已经是该组织的成员”“Access denied”等。

### 上传相关约束
- 非法文件上传场景需至少覆盖非图片文件。
- 上传目录应在测试后清理，避免残留文件影响后续执行。
- 若要覆盖大小限制，后续测试用例需明确构造超限文件，但本设计阶段先将其保留为可选扩展项。

5. 已形成清晰的测试结果记录，说明通过项、失败项与剩余风险。

## 风险与权衡
- 若当前产品的组织加入流程入口不稳定，相关集成测试需要先做一次页面侦察和选择器稳定性确认。
- 多用户浏览器测试需要处理会话隔离，建议使用不同浏览器上下文或显式退出再登录。
- 当前仓库缺少统一测试数据重置机制，测试前需要明确数据库清理策略。
- 若一次性自动化过多低频场景，会明显拖慢落地速度，因此本次接受“完整用例 + 核心自动化 + 其余手工补充”的权衡。
